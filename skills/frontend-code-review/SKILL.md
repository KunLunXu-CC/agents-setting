# 前端代码审查

## 目标

当用户要求审查前端代码（尤其是 `.tsx`、`.ts`、`.js` 文件）时使用此 skill。支持两种审查模式：

1. **待提交变更审查** - 检查暂存区/工作区中计划提交的文件, 在提交前标记违反清单的项。
2. **指定文件审查** - 审查用户点名的文件, 并报告对应清单中的问题。

对每个适用文件和模式, 都必须严格遵循下方清单。

## 检查清单

- [references/code-quality.md](references/code-quality.md)
- [references/performance.md](references/performance.md)
- [references/business-logic.md](references/business-logic.md)

每条规则违规都要标注紧急度元信息, 方便后续评审者排序修复优先级。

## 审查流程

1. 打开相关组件/模块, 收集与类名、React Flow hooks、props 记忆化、样式实现相关的代码行。
2. 对每条审查规则, 记录代码偏离点并截取有代表性的片段。
3. 按下方模板撰写审查结果。先按 **Urgent** 分组, 再按类别顺序（Code Quality、Performance、Business Logic）排列。

## 必须输出格式

执行该 skill 时, 回复必须严格使用以下两种模板之一：

### 模板 A（有发现）

```
# 代码审查
发现 <N> 个必须修复的紧急问题：

## 1 <问题简述>
FilePath: <path> line <line>
<相关代码片段或定位信息>


### 建议修复
<修复建议简述>

---
...（每个紧急问题重复一次）...

发现 <M> 条改进建议：

## 1 <建议简述>
FilePath: <path> line <line>
<相关代码片段或定位信息>


### 建议修复
<修复建议简述>

---

...（每条建议重复一次）...
```

如果没有紧急问题, 省略对应部分；如果没有改进建议, 也省略对应部分。

如果问题数量超过 10, 汇总为“10+ 个紧急问题”或“10+ 条改进建议”, 仅输出前 10 条。

不要压缩各段之间的空行, 按原样保留以确保可读性。

如果使用模板 A（即存在需修复问题）, 且至少有 1 个问题需要修改代码, 则在结构化输出后附上一句简短追问, 询问用户是否需要你按“建议修复”直接修改。例如：“是否要我根据‘建议修复’部分直接处理这些问题？”

### 模板 B（无问题）

```
## 代码审查
未发现问题。
```
